# WBS 作成ガイド

## 分解の基本ルール

### 粒度の目安
- **良い**: 「Supabaseにusersテーブルを作成しRLSを設定する」（4h以内）
- **悪い**: 「バックエンドを作る」（大きすぎ）
- **悪い**: 「変数名をsnake_caseに変える」（小さすぎ・実装中に勝手にやる）

### MVP の判断基準
「これだけあれば価値を確認できる」という最小構成。
- UIが美しくなくてもOK
- エラーハンドリングが最低限でもOK
- ただし「動作する」は必須

### 依存関係の書き方
- `依存: なし` — 他に依存しない先頭タスク
- `依存: #1` — タスク1が完了してから着手
- `依存: #1, #3` — 複数に依存

### 完了条件（DoD）の書き方
- 動詞で始める：「〜ができる」「〜が返る」「〜が表示される」
- 測定可能に：「テストが通る」「ログに出力される」「UIに表示される」
- あいまいな表現を避ける：「動く」「問題ない」はNG

## フェーズ分けの判断

| フェーズ | 含めるもの |
|---|---|
| MVP | コア機能のみ。最速で価値を確認できる最小セット |
| Phase 1 | エラーハンドリング、バリデーション、本番品質のUI |
| Phase 2 | 拡張機能、パフォーマンス改善、Analytics |

## タスク漏れを防ぐチェックリスト

実装系タスクの他に以下が抜けやすい：
- [ ] 環境変数の設定（.env）
- [ ] デプロイ設定
- [ ] RLS / 認証設定（Supabaseの場合）
- [ ] エラーハンドリング（Phase 1でOK）
- [ ] ローカル動作確認の手順
- [ ] README / CLAUDE.md の更新

## 見積もりの目安

| 作業 | 目安 |
|---|---|
| Supabaseテーブル作成 + RLS | 1-2h |
| FastAPI エンドポイント1本 | 1-3h |
| 基本的なUIコンポーネント1つ | 2-4h |
| 外部API連携1本（既存SDK使用） | 2-4h |
| 認証フロー（Supabase Auth） | 4-8h |
| デプロイ設定（初回） | 2-4h |
