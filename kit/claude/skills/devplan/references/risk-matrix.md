# リスク分類表

## リスクカテゴリ

### 技術リスク
| リスク | 典型例 | 影響度 | 対策 |
|---|---|---|---|
| 未検証の技術スタック | 初めて使うライブラリ | 中〜高 | PoC（概念実証）を先行させる |
| API制限・レート制限 | OpenAI, LINE, Supabase | 中 | Sandbox環境で先に確認 |
| パフォーマンス問題 | 大量データ処理 | 中 | 早期のベンチマーク |
| 外部サービス障害 | 依存APIのダウン | 低〜中 | フォールバック / リトライ設計 |

### 要件リスク
| リスク | 典型例 | 影響度 | 対策 |
|---|---|---|---|
| 仕様未確定 | 「とりあえず作って」 | 高 | プランフェーズで必ず確認 |
| スコープクリープ | 実装中の追加要望 | 高 | プラン外は次フェーズに記録 |
| 認識ズレ | 完成イメージの差異 | 高 | プラン承認で合意取り |

### セキュリティリスク（Supabaseプロジェクト必須）
| リスク | 影響度 | 対策 |
|---|---|---|
| RLS設定漏れ | 高 | 全テーブルにRLS必須 |
| secret露出 | 高 | `NEXT_PUBLIC_` に秘密キー絶対禁止 |
| 他人データアクセス | 高 | 所有者チェック必須 |

### スケジュールリスク
| リスク | 典型例 | 影響度 | 対策 |
|---|---|---|---|
| 見積もり甘い | 「簡単そう」が難航 | 中 | バッファ20〜50%を加算 |
| 依存タスクの遅延 | 外部API Key待ち | 中 | 並行できるタスクを先行 |

## 影響度の判定基準

| 影響度 | 基準 |
|---|---|
| 高 | プロジェクトが止まる / セキュリティ問題 |
| 中 | 一部機能が使えない / 遅延が生じる |
| 低 | 軽微な不便 / 代替手段がある |

## 前提条件の書き方

前提条件 = 「これが崩れたら設計を見直すトリガー」

例：
- 「Supabase の無料枠で要件を満たせる（行数・ストレージ・Transfer量）」
- 「LINE Messaging API が現行仕様のまま変わらない」
- 「ユーザー数が月1,000以下に留まる（スケール前提なし）」
