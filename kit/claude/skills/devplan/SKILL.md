# DevPlan — 実装前プランニングスキル

## このスキルの目的

「作る前に考える」を徹底することで、実装中の手戻り・設計ミス・認識ズレをゼロにする。
プランが承認されるまでコードは一切書かない。

---

## 実行フロー（全7フェーズ）

### フェーズ0: コンテキスト収集（自動）

スキル起動直後に以下を自動で確認する：

```bash
# プロジェクト固有ルールの確認
cat CLAUDE.md 2>/dev/null || echo "CLAUDE.md なし（グローバルルールで進行）"

# 既存コードベースの把握（関連ファイルのみ）
ls -la 2>/dev/null | head -20

# 過去のプランがあれば参照
ls docs/plans/ 2>/dev/null || true
```

収集した情報をもとに、ユーザーへの質問を最小化する。

---

### フェーズ1: 目的・要件の確認

以下を1問ずつ聞く（すでに情報がある項目はスキップ）：

1. **何を作る / 何を解決したいか？**（1〜3文で）
2. **誰が使うか？**（エンドユーザー、内部ツール、APIなど）
3. **成功の定義は何か？**（「これができれば完成」の基準）
4. **制約・締め切りはあるか？**（期日、技術スタック縛り、予算など）
5. **やらないこと（スコープ外）は何か？**

---

### フェーズ2: 技術選定の確認

既存スタックがある場合はそれを踏襲。新規の場合のみ提案する。

確認項目：
- 言語・フレームワーク
- データストア（DB / ファイル / API）
- 外部サービス依存（Supabase / Vercel / LINE / etc.）
- デプロイ先

**例：あなたの標準スタックをここに定義**（プロジェクトのデフォルト）：
- バックエンド: （例）Python 3.9+ / FastAPI
- DB: （例）Supabase (PostgreSQL + RLS)
- フロント: （例）Next.js（必要時のみ）
- デプロイ: （例）Vercel

---

### フェーズ3: アーキテクチャ設計

コンポーネント図を ASCII で描く（複雑な場合のみ）。

必ず以下を明示：
- データの流れ（入力 → 処理 → 出力）
- 外部依存の境界線
- 認証・認可の境界

---

### フェーズ4: 工程分解（WBS）

タスクを以下のルールで分解：

- **最小単位**: 1〜4時間で完結するもの
- **依存関係**: 先行タスクを明示
- **フェーズ分け**: MVP / Phase1 / Phase2 の3層で優先度を付ける
- **各タスクに**: 完了条件（DoD）を1行で書く

フォーマット：
```
## MVP（最小動作版）
- [ ] タスク名｜依存: なし｜DoD: XXXができること｜見積: Xh
- [ ] タスク名｜依存: 上記｜DoD: YYYができること｜見積: Xh

## Phase 1（本番品質）
- [ ] タスク名｜依存: MVP完了｜DoD: ZZZができること｜見積: Xh
```

---

### フェーズ5: リスク・前提条件の洗い出し

| カテゴリ | 内容 | 影響度 | 対策 |
|---|---|---|---|
| 技術リスク | 未検証のライブラリ・API | 中 | PoC先行 |
| 要件リスク | 仕様が未確定な部分 | 高 | 事前合意 |
| 外部依存 | サードパーティの制限 | 低〜高 | フォールバック設計 |

前提条件（これが崩れたら設計から見直し）も列挙する。

---

### フェーズ6: プラン文書の生成・保存

プランを `docs/plans/YYYY-MM-DD_[プロジェクト名].md` に保存する。

```bash
mkdir -p docs/plans
# ファイル名例: docs/plans/2026-05-07_myproject-phase2.md
```

テンプレートは `references/plan-template.md` を使用。

---

### フェーズ7: 承認フェーズ（必須）

プラン文書をユーザーに提示し、以下を確認する：

```
プランを作成しました。

[プラン概要をここに表示]

承認しますか？
  [A] このまま実装開始
  [B] 一部修正してから実装
  [C] 設計を根本から見直す
```

**ユーザーの「OK」または「A」が確認されるまで実装ステップに移らない。**

---

## 出力フォーマット（必須構成）

プラン文書には必ず以下のセクションを含める：

```markdown
# [プロジェクト名] 実装プラン

## 概要
- 目的:
- 対象ユーザー:
- 完成の定義:
- スコープ外:

## 技術スタック
| 役割 | 選定技術 | 理由 |
|---|---|---|

## アーキテクチャ
[データフロー図 / コンポーネント図]

## 実装工程（WBS）
### MVP
### Phase 1
### Phase 2（任意）

## リスク・前提条件

## 未解決事項（Open Questions）

## 承認欄
- [ ] ユーザー承認: YYYY-MM-DD
```

---

## ルール（厳守）

1. **プランなしの実装禁止**: このスキルが完了するまでコードを書かない
2. **承認なしの着手禁止**: フェーズ7でOKが出るまで実装に移らない
3. **スコープクリープ防止**: プラン外の追加要求は「次のフェーズ」として記録し、今回のスコープには含めない
4. **推測で進まない**: 不明点はフェーズ1で必ず聞く。実装中に戻って聞かない
5. **プランは更新する**: 実装中に設計変更が必要になったら、コードより先にプランを更新する

---

## 起動メッセージ

スキル起動時に必ず以下を出力する：

```
DevPlan 起動。実装前のプランニングを開始する。

何を作るか教えてほしい。できるだけ具体的に（背景・目的・誰が使うか）。
```

---

## 連携スキル

- **verify**: プラン承認後、実装完了時の検証に使用
- **/code-review**（組み込みコマンド）: 実装後のコードレビューに使用
- DB初期化・リポジトリ作成・デプロイなどプロジェクト固有の定型作業は、`skill-creator-max` で自作スキル化して追加する
